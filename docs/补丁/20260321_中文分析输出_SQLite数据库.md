# 补丁：中文分析输出 + PostgreSQL 数据库存储

**日期**：2026-03-21 ~ 2026-03-22
**范围**：后端 + 前端 + Agent 层

---

## 一、变更概述

本次补丁包含三个核心功能：

1. **PostgreSQL 数据库替代 JSON 文件存储** — 用量数据、分析记录、用户信息持久化到远程 PostgreSQL（`34.133.47.117`），psycopg2 连接池支持高并发；未配置 DATABASE_URL 时自动降级为本地 SQLite
2. **分析报告中文输出** — 12 个 Agent 的 prompt 注入中文指令后缀，Web 端默认以中文输出分析报告
3. **中文指令增强** — 将中文指令置于 system prompt 最末尾 + 角色设定 + 双语强调，提升 LLM 遵循度

## 二、新增文件

### `web/database.py`

统一数据库封装模块，支持 PostgreSQL / SQLite 双后端。

- **后端选择**：`DATABASE_URL` 环境变量存在时用 PostgreSQL，否则降级为 SQLite
- **PostgreSQL 连接池**：`psycopg2.pool.ThreadedConnectionPool(min=2, max=10)`
- **表名前缀**：`trading_`（在 agentpit 共享数据库中隔离）
- **表结构**：
  - `trading_users` — 用户信息（user_id PK, name, email, avatar）
  - `trading_analysis_records` — 分析记录（SERIAL PK, report_json 存完整报告）
  - 索引：`idx_trading_records_user(user_id, timestamp)`
- **自动迁移链**：JSON → DB，SQLite → PostgreSQL（启动时一次性执行）

**核心方法**：

| 方法 | 说明 |
|------|------|
| `upsert_user(user_id, name, email, avatar)` | 创建或更新用户，非空字段覆盖 |
| `record_analysis(user_id, ticker, ..., report)` | 插入分析记录，PG 用 `RETURNING id` |
| `get_history(user_id)` | 返回用户历史列表（不含 report_json），按时间倒序 |
| `get_record(record_id, user_id)` | 返回单条记录含解析后的 `report` dict |
| `get_usage_summary(user_id)` | SQL 聚合：总计 + 月度分组 + 最近 50 条 |

### 表结构

```sql
-- 在 agentpit 数据库中，trading_ 前缀隔离
CREATE TABLE trading_users (
    user_id TEXT PRIMARY KEY,
    name TEXT, email TEXT, avatar TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE trading_analysis_records (
    id SERIAL PRIMARY KEY,
    user_id TEXT NOT NULL REFERENCES trading_users(user_id),
    ticker TEXT NOT NULL,
    trade_date TEXT NOT NULL,
    decision TEXT,
    tokens_in INTEGER DEFAULT 0,
    tokens_out INTEGER DEFAULT 0,
    llm_calls INTEGER DEFAULT 0,
    tool_calls INTEGER DEFAULT 0,
    elapsed_ms INTEGER DEFAULT 0,
    model TEXT,
    report_json TEXT,
    timestamp TEXT NOT NULL
);
CREATE INDEX idx_trading_records_user ON trading_analysis_records(user_id, timestamp);
```

### 后端切换逻辑

```
DATABASE_URL 存在？
  ├─ 是 → psycopg2 连接池 (ThreadedConnectionPool 2-10)
  └─ 否 → 本地 SQLite data/trading.db (WAL 模式, 开发用)
```

## 五、数据迁移

启动时自动执行三级迁移链：

1. **JSON → DB**：检测 `data/web_usage.json`，逐用户插入 `trading_users`（`ON CONFLICT DO NOTHING`），逐条插入 `trading_analysis_records`，完成后重命名为 `.json.bak`
2. **SQLite → PostgreSQL**（仅 PG 后端）：检测 `data/trading.db`，若 PG 表为空则全量导入用户和记录，完成后重命名为 `.db.bak`
3. 已迁移的文件不会重复处理

> **修复记录**：首次部署时 FK 约束失败（JSON 中 user_id 未在 trading_users 中），已在迁移前自动创建用户占位。

## 六、中文指令增强（二次修复）

首次部署后发现分析报告仍为英文。原因：

1. 4 个 Analyst 的 `{system_message}` 被嵌在 system prompt 中间，中文指令不在末尾
2. 中文后缀措辞单薄，对 Gemini 模型约束力不够

**修复**：
- `language.py`：增强为角色设定 + Markdown 标题 + 双语强调
- 4 个 Analyst：`lang_suffix` 改为 `{lang_suffix}` 模板变量，`prompt.partial(lang_suffix=...)` 注入到 system prompt **最末尾**
